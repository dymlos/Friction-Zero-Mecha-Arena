# Proximos pasos

La siguiente iteracion recomendada debe tomar el loop integrado actual y `Modo Practica` como nueva base cerrada: shell, match y practica ya quedaron revalidados de forma automatizada. Lo siguiente ya no es definir practica, sino validarla con evidencia humana y decidir si el siguiente empuje real es `M9` o un polish puntual muy acotado.

## Prioridad inmediata

1. Ejecutar smoke manual de `Practica` con foco en legibilidad, ritmo y descubrimiento real:
   - `1P Easy`
   - `1P Hard`
   - `2P mixto Easy/Hard`
   - entrada desde `How to Play`
   - volver al menu desde pausa
2. Revalidar manualmente shell + practica en shared-screen con jugadores reales antes de abrir settings amplios o polish audiovisual.
3. Mantener ownership estricto: `Characters` = identidad, `How to Play` = reglas base, `Practica` = experimentacion segura, `match` = decision tactica bajo presion.
4. Mantener la disciplina de paridad `base/validation` mientras se toque shell, practica o HUD.
5. Con esa evidencia, decidir si el siguiente bloque es `M9` (configuracion/UX operativa) o un polish puntual de learnability.

## Diferidos explicitos de este cierre parcial

- replay snippets
- post-muerte `FFA`
- extensiones amplias de shell
- expansion futura de roster

## Regla de sesion

- Si la sesion es documental, priorizar ownership de superficies y evitar duplicar copy entre `Characters`, `How to Play`, practica y match.
- Si la sesion es tecnica, no perder disciplina de paridad `base/validation` ni reintroducir metadata de laboratorio en la ruta de jugador.
- Si la sesion es de gameplay, no abrir polish audiovisual ni balance fino antes del smoke manual de practica y el siguiente bloque de playtests.
